1. Vue d'ensemble

Contexte métier, rôles et flux de validation.

1.1 Objectif de l'application

Boost Manager est une plateforme de gestion de campagnes publicitaires Meta/Facebook pour les équipes Media Buyer. L'application permet aux opérateurs de créer des demandes de boost sur des posts Facebook, de les soumettre à une chaîne de validation N+1 → N+2, puis de déclencher la création automatique de campagnes sur Meta Ads via N8N.

Principe fondamental : Chaque utilisateur (opérateur, validateur) n'est visible que sur les pages Facebook qui lui sont assignées. Seul l'admin voit tout.

1.2 Rôles utilisateurs

Rôle Identifiant Capacités principales
Admin admin Accès total — gestion utilisateurs, pages, paramètres N8N/Meta, bypass validation
Opérateur operator Crée des campagnes sur ses pages assignées, soumet pour validation, lance le boost, active sur Meta
Validateur N+1 validator_n1 Approuve ou rejette les campagnes en pending_n1 sur ses pages assignées
Validateur N+2 validator_n2 Approuve ou rejette les campagnes en pending_n2 sur ses pages assignées
Validateur validator Cumule N+1 + N+2 (rôle polyvalent)

3.2 Authentification (Laravel Sanctum)

L'app mobile utilise des tokens Sanctum (Personal Access Tokens). Chaque appel API doit inclure l'en-tête :

Authorization: Bearer {token}
Accept: application/json
Content-Type: application/json

À implémenter dans Laravel :

  • Installer Sanctum : php artisan vendor:publish --provider="Laravel\Sanctum\SanctumServiceProvider"
  • Ajouter colonne fcm_token dans la table users pour les push notifications
  • Créer le groupe de routes api avec prefix /api/v1
  • Retourner toujours du JSON avec le format uniforme ci-dessous

Format de réponse uniforme

// Succès
{
  "success": true,
  "data": { /* objet ou tableau */ },
  "meta": { "current_page": 1, "last_page": 5, "total": 48 } // pagination si applicable
}

// Erreur
{
  "success": false,
  "message": "Message d'erreur lisible",
  "errors": { "field": ["validation error"] } // optionnel, erreurs de validation
}

3.3 Base de données partagée

Principe : L'application mobile n'a pas sa propre base de données. Elle accède exactement à la même base MySQL que l'application web via les API Laravel. Aucune migration supplémentaire n'est nécessaire, sauf l'ajout de la colonne fcm_token.

Migration à ajouter côté Laravel :

Schema::table('users', function (Blueprint $table) {
    $table->string('fcm_token')->nullable(); // Firebase push token
});

5. API REST

Tous les endpoints sont préfixés /api/v1. Les endpoints protégés nécessitent le header Authorization: Bearer {token}.

5.1 Authentification

POST /api/v1/auth/login — Connexion, retourne le token Sanctum
Body : email, password, fcm_token (optionnel, stocker le token Firebase)
Réponse : token (string), user (objet avec id, name, email, roles, avatar)
POST /api/v1/auth/logout — Révoke le token courant Auth
GET /api/v1/auth/me — Profil utilisateur + rôles + pages assignées Auth
Réponse : user object avec roles (tableau), facebook_pages (pages assignées), unread_notifications_count
PUT /api/v1/auth/fcm-token — Met à jour le token Firebase push Auth
Body : fcm_token (string, required)

5.2 Dashboard

GET /api/v1/dashboard — KPIs + campagnes récentes Auth
Réponse : 
// Filtré selon le rôle et les pages assignées
{
  "counts": {
    "total": 42, "draft": 5, "pending_n1": 3, "pending_n2": 2,
    "approved": 8, "done": 20, "error": 4
  },
  "recent_campaigns": [/* 5 dernières BoostCampaign */],
  "pending_validation_count": 5, // pour les validateurs
  "last_sync": { "status": "FINISHED", "started_at": "..." } // pour admin/validators
}

5.3 Posts Facebook

GET /api/v1/posts — Liste posts boostables (filtrés par pages assignées) Auth
Query params : page_id (filtre page), type (photo/video/link), search, per_page (défaut 20)
Réponse : pagination avec items contenant : post_id, message, thumbnail_url, permalink_url, type, posted_at, is_boostable, page (nom de page)

5.4 Campagnes Media Buyer

GET /api/v1/campaigns — Liste campagnes (filtrées par rôle + pages) Auth Tous rôles
Query params : status (filtre statut), per_page, page
Réponse : pagination avec les champs de boost_campaigns + status_label + budget_formatted + user (nom opérateur pour les validators)
GET /api/v1/campaigns/{id} — Détail d'une campagne Auth
Réponse : campagne complète + status_label + budget_formatted + user + post (thumbnail, message)
POST /api/v1/campaigns — Créer une campagne (brouillon) Auth operator, admin
Body (tous requis sauf mention) : 
{
  "post_id": "123456789_987654321",
  "campaign_name": "Campagne Mars 2026",
  "campaign_objective": "OUTCOME_TRAFFIC",
  "special_ad_categories": "NONE",
  "campaign_status": "PAUSED",
  "existing_campaign_id": null, // optionnel
  "adset_name": "Ad Set Mars 2026",
  "budget_type": "lifetime_budget",
  "budget_value": 50000, // en FCFA, min 500
  "duration_days": 7, // 1–90
  "countries": ["CI", "SN"],
  "interests": ["Football", "Musique"], // optionnel
  "optimization_goal": "LINK_CLICKS",
  "billing_event": "IMPRESSIONS",
  "bid_strategy": "LOWEST_COST_WITHOUT_CAP",
  "ad_name": "Annonce Mars 2026",
  "ad_status": "PAUSED"
}
POST /api/v1/campaigns/{id}/submit — Soumettre pour validation N+1 Auth operator (propriétaire), admin
POST /api/v1/campaigns/{id}/approve — Approuver (N+1 → pending_n2, N+2 → approved) Auth validator_n1, validator_n2, validator, admin
POST /api/v1/campaigns/{id}/reject — Rejeter avec motif Auth validator_n1, validator_n2, validator, admin
Body : reason (string, requis, max 500 chars)
POST /api/v1/campaigns/{id}/launch — Déclencher le boost via N8N (approved → running) Auth operator (propriétaire), admin
POST /api/v1/campaigns/{id}/activate — Activer sur Meta (paused_ready → active) Auth operator (propriétaire), admin

5.5 File de validation

GET /api/v1/campaigns/pending — Campagnes en attente de ma validation Auth validator_n1, validator_n2, validator, admin
N+1 voit pending_n1, N+2 voit pending_n2, admin/validator voient les deux. Filtré par pages assignées.

5.6 Notifications

GET /api/v1/notifications — Liste notifications (non lues en premier) Auth
Réponse : data (liste), unread_count (int).
Chaque notification : id, type, data.message, data.boost_id, read_at, created_at
POST /api/v1/notifications/{id}/read — Marquer une notification comme lue Auth
POST /api/v1/notifications/read-all — Tout marquer comme lu Auth

5.7 Pages Facebook

GET /api/v1/pages — Pages Facebook accessibles à l'utilisateur Auth
Admin : toutes les pages actives. Autres : pages assignées uniquement.
Réponse : id, page_id, page_name, is_active (pas de access_token)

Les notifications push complètent les notifications in-app via Firebase FCM.

8.2 Gestion des erreurs réseau

  • 401 Unauthorized : Supprimer le token local → rediriger vers Login
  • 403 Forbidden : Toast "Vous n'avez pas la permission d'effectuer cette action"
  • 422 Unprocessable Entity : Afficher les erreurs de validation champ par champ
  • 500 Server Error : Toast "Erreur serveur, veuillez réessayer"
  • Pas de réseau : Mode hors-ligne limité — afficher les données en cache, désactiver les actions

8.3 Sécurité

  • Stocker le token Sanctum dans le Keychain (iOS) / Keystore (Android) — jamais en AsyncStorage/SharedPreferences non chiffré
  • Le access_token Meta n'est jamais exposé via l'API — il reste côté serveur
  • HTTPS obligatoire en production
  • Rafraîchir le fcm_token à chaque démarrage de l'app

8.4 Filtrage par page

Important : Le filtrage par page assignée est entièrement géré côté serveur par Laravel. L'app mobile n'a pas besoin de filtrer elle-même — les API retournent déjà uniquement les données autorisées pour l'utilisateur connecté. L'admin voit tout, les autres voient uniquement leurs pages assignées.
